# Email Agent - 개발 방침

> 이 문서는 오래 유지할 원칙만 적는다. 현재 상태와 세부 진행 상황은 `docs/status.md`를 기준으로 관리한다.

## 0. 작업 모드

### 0-1. 협업 모드

- 기본 모드는 `협업 모드`다.
- 사용자의 중간 질문, 선택, 피드백을 자주 반영하면서 진행한다.
- 이 모드에서도 모든 비사소한 작업 단위의 첫 게이트는 `docs/AGENT.md`다.
- 다음 큰 단계로 넘어갈 때는 사용자의 현재 의도를 더 자주 확인하는 편으로 운영한다.
- 이 모드에서 assistant는 사용자에게 보내는 답변 앞에 기본적으로 `협업)` prefix를 붙인다.

### 0-2. 에이전트 모드

- 사용자가 명시적으로 `에이전트 모드`를 선언하면, 그 시점부터는 문서에 기록된 목표와 규칙을 기준으로 스스로 이어서 진행한다.
- 이 모드에서는 사용자가 자리를 비웠을 가능성을 기본 전제로 둔다.
- 명시적인 중단 지시가 없는 한 이미 시작한 수집, 동기화, 백필, 분류, 보고, 배포 준비 작업을 끊지 않는다.
- 장시간 다단계 파이프라인은 가능하면 현재 대화 세션이나 단일 터미널에만 의존하지 않게 만든다.
- `10분 이상` 걸릴 가능성이 있거나 `여러 단계가 자동으로 이어지는` 작업은 watchdog 대상으로 본다.
- 이 모드에서 assistant는 사용자에게 보내는 답변 앞에 기본적으로 `AI)` prefix를 붙인다.

### 0-3. 안내 말투

- 사용자에게 설명하거나 진행 상황을 알릴 때는 친절하고 차분한 말투를 유지한다.
- 다만 친절함 때문에 판단이나 상태를 흐리게 쓰지 않는다.
- 즉 톤은 부드럽게 유지하되, 현재 상태, 위험, 다음 단계는 분명하게 전달한다.

## 1. 프로젝트 방향

- 목표는 이메일을 받아 구조화된 업무 결과로 바꾸는 재사용 가능한 자동화 스택을 만드는 것이다.
- 현재 구현 순서는 `mailbox -> analysis -> exports`다.
- 초반 기본 전달물은 `이메일 수신 -> 분류/추출 -> Excel 출력` 경로다.
- 답장 자동화와 외부 알림은 위 기본 경로가 안정화된 뒤 점진적으로 붙인다.

## 2. 핵심 개발 원칙

### 2-1. 구조화 출력 우선

- LLM 응답은 자유서술보다 schema 기반 구조화 출력을 우선한다.
- 분류, 추출, 요약, 답장 draft 모두 가능한 한 명시적 필드 계약을 먼저 정한다.

### 2-2. idempotent 처리 우선

- 같은 이메일을 여러 번 읽어도 중복 row, 중복 답장, 중복 알림이 생기지 않는 구조를 우선한다.
- mailbox cursor, message id, thread id, export row key, action log를 일관된 기준으로 남긴다.

### 2-3. 사람 승인 경계는 명확하게 둔다

- 자동 reply나 외부 발송 기능은 기본적으로 `draft -> review -> send` 경계를 먼저 만든다.
- 초기 단계에서는 `무조건 자동 발송`보다 `안전한 draft 생성`을 우선한다.

### 2-4. 민감정보와 원문 보호

- 실제 이메일 원문, 첨부파일, API 키, 계정 정보는 공개 문서에 적지 않는다.
- 리포 바깥 `../secrets/`와 `../results/` 계층을 사용해 로컬 전용 데이터와 실행 산출물을 분리한다.

### 2-5. 단순한 모듈 인터페이스

- 각 요소기술은 독립 폴더와 공통 진입점으로 관리한다.
- provider나 backend가 여러 개여도 상위 사용법은 하나로 유지한다.
- 데모, 실험, 운영 코드는 가능한 한 같은 파일에 섞지 않는다.

### 2-6. 오버엔지니어링 금지

- 처음부터 과도한 추상화, 미사용 옵션, 추측성 fallback을 넣지 않는다.
- 먼저 주경로를 단순하게 완성하고, 실제 문제를 본 뒤 필요한 만큼만 확장한다.

### 2-7. 코드 스타일 단순화

- 함수 시그니처는 현재 단계에서 지나치게 복잡하게 만들지 않는다.
- 모든 함수는 한국어 docstring으로 `기능 / 입력 / 반환`을 기본적으로 적는다.

### 2-8. 계획 기반 실행

- 작업은 시작 전에 짧은 계획을 세우고 순서대로 진행한다.
- 중간에 계획이 바뀌면 이유를 남긴다.
- 장시간 실행, 다단계 자동화, 대량 백필처럼 나중에 컨텍스트가 많이 압축될 수 있는 작업은 실행 전에 계획 문서를 남긴다.

### 2-9. 장시간 실행 로그

- 1분 이상 걸릴 수 있는 실행은 진행 상태를 주기적으로 출력한다.
- 재실행이 비싼 작업은 가능하면 부분 재실행과 중간 상태 확인을 지원한다.
- 긴 명령이 돌 때는 가능한 범위에서 핵심 터미널 출력도 함께 요약해 사용자에게 알린다.
- watchdog를 붙일 장시간 작업은 실전 실행 전에 sample 코드와 smoke 결과를 먼저 만든다.

### 2-10. 디렉토리 구조도 주기적으로 정리한다

- 코드, 문서, 템플릿, 예시, 생성 결과가 한 디렉토리에 뒤섞이지 않도록 주기적으로 구조를 점검한다.
- 새 파일을 만들 때도 처음부터 `런타임 코드`, `실험 코드`, `도구`, `문서`, `예시`, `생성 산출물`의 역할을 구분한다.
- 아직 구현을 시작하지 않은 미래 기능용 빈 디렉토리와 플레이스홀더 README는 미리 늘리지 않는다.
- 공용 모듈도 실제로 둘 이상의 활성 경로가 공유할 때만 추출한다.

### 2-11. 문서 refresh를 작업 시작 게이트로 둔다

- 비사소한 작업에서는 시작 게이트를 반드시 다시 통과한다.
- 특히 `에이전트 모드`에서는 새 단계 착수 전, 결과 정리 직전, 후속 실행 시작 직전에 다시 `docs/README.md`와 `docs/status.md`를 읽는다.

### 2-12. 진행 보고는 짧고 자주 한다

- 장시간 작업에서는 진행 보고 빈도를 낮추지 않는다.
- 보고 내용은 기본적으로 `무엇을 확인했는지 / 어디서 막혔는지 / 다음에 무엇을 할지`를 포함한다.
- 사용자는 터미널 원문을 보지 못하므로, 긴 실행에서는 핵심 출력과 의미를 짧게 번역해 같이 전달한다.
- 실제로 돌고 있는 작업을 말할 때는 반드시 `ps`, `tty`, 진행률 파일, 상태 파일 같은 근거를 먼저 확인한다.
- 이런 진행 보고도 친절한 안내 말투를 유지하되, 사실 관계와 판단 기준은 직접적으로 적는다.

### 2-13. 장시간 작업은 마일스톤 단위로 commit 여부를 점검한다

- 장시간 작업은 끝날 때만 한 번에 정리하지 말고, 안정된 마일스톤마다 local commit 필요 여부를 다시 판단한다.
- push는 자동 상시 동작으로 두지 않고, 원격에 남길 가치가 있는 마일스톤인지 판단한 뒤 진행한다.
- commit 메시지는 기본적으로 한국어 중심으로 쓴다.

### 2-14. background job은 detached 기준으로 본다

- polling, backfill, bulk export, 큐 worker처럼 장시간 도는 작업은 단순한 포그라운드 셸 루프보다 `pid`, `status`, `events`를 남기는 detached supervisor를 기본값으로 본다.
- 장시간 작업의 최소 산출물은 `job spec`, `pid`, `status.local.md`, `events.log`, `stdout/stderr`, `종료 코드 또는 종료 원인`이다.

## 3. 문서 운영 원칙

### 3-1. 문서 역할은 하나씩만 가진다

| 문서 | 역할 |
|---|---|
| `README.md` | 레포 소개와 진입점 |
| `docs/README.md` | 문서 허브 |
| `docs/status.md` | 현재 상태와 다음 작업의 단일 기준 |
| `docs/decisions.md` | 현재 유효한 핵심 결정 |
| `docs/logbook.md` | 최근 작업 로그 |
| `docs/archive/` | 오래된 상세 기록 |

- 같은 상태 설명을 여러 상위 요약 문서에 반복하지 않는다.
- 상태가 바뀌면 우선 `docs/status.md`를 갱신한다.
- 원칙이 바뀌었을 때만 `docs/개발방침.md`를 갱신한다.
- 새 에이전트나 새 세션의 시작 절차는 `docs/README.md`를 기준으로 통일한다.

### 3-2. 상위 문서는 얇게 유지한다

- 루트 `README.md`에는 프로젝트 소개, 모듈 링크, 문서 진입점만 둔다.
- `docs/README.md`는 색인과 역할 설명만 둔다.
- 현재 단계에서 보조 개요 문서는 꼭 필요할 때만 추가한다.

### 3-3. 결정과 로그는 분리한다

- `docs/decisions.md`에는 현재도 유효한 프로젝트 결정만 남긴다.
- 세부 과정, 시행착오, 실행 기록은 `docs/logbook.md` 또는 `docs/archive/`로 보낸다.

### 3-4. 최근 로그만 활성 문서에 둔다

- `docs/logbook.md`는 최근 세션만 유지한다.
- 문서가 길어지거나 현재 단계와 직접 관련 없는 항목이 많아지면 `docs/archive/`로 옮긴다.

### 3-5. 생성 산출물과 사람이 유지하는 문서를 분리한다

- code-generated CSV, JSON, XLSX, HTML, 로그 파일은 공식 문서가 아니다.
- 자동 생성 결과물은 리포 바깥 결과 디렉토리에 두고, 사람이 계속 읽을 필요가 있는 경우에만 요약을 문서로 승격한다.

### 3-6. 환경 문서는 별도 유지한다

- 설치 절차, venv 경로, 검증 명령은 관련 모듈 문서에서 관리한다.
- 상위 문서에는 환경 문서 링크만 두고 설치 명령을 반복하지 않는다.

### 3-7. 민감 정보는 공개 문서에 쓰지 않는다

- 실제 계정, 키, 메일 원문, 첨부파일, 내부 주소, 사용량 로그 같은 운영 정보는 리포 바깥 로컬 문서에서만 관리한다.

### 3-8. 누적 보고서 문서는 날짜 prefix로 정렬 가능하게 만든다

- `README`, `status`, `개발방침`, `decisions`, `logbook`처럼 항상 최신 기준을 가리켜야 하는 문서는 고정 파일명을 유지한다.
- 누적되는 조사/환경/보고서는 `YYMMDD_HHMM_한국어제목.md` 형식을 기본으로 쓴다.

## 4. 자산과 데이터 운영 원칙

- 실제 고객 메일, 대용량 첨부파일, 임시 export, 로컬 DB는 기본적으로 리포에 포함하지 않는다.
- 다만 재현에 꼭 필요한 소형 예제나 redacted sample은 예외적으로 포함할 수 있다.
- 문서용 스크린샷, 다이어그램, 샘플 이미지는 `docs/assets/` 아래에서 정리한다.

### 4-1. 로컬 워크스페이스 구조

- 로컬 워크스페이스는 한 루트 아래 `repo/`, `envs/`, `results/`, `secrets/`를 sibling으로 둔다.
- `repo/`는 Git 추적 대상 코드와 문서만 둔다.
- `envs/`는 로컬 가상환경만 둔다.
- `results/`는 재생성 가능한 실행 결과와 임시 프로파일을 둔다.
- `secrets/`는 Git에 넣을 수 없는 로컬 전용 운영 정보만 둔다.

### 4-2. 추적 자산과 로컬 자산을 구분한다

- 작은 템플릿이나 redacted sample은 리포에 포함할 수 있다.
- 실제 메일 데이터, 첨부파일 원본, 대량 export 결과, 인증 캐시는 로컬 경로 또는 gitignore 경로에 둔다.

## 5. 문서를 갱신할 때의 체크리스트

1. 이번 변경의 현재 상태 기준이 바뀌었는지 먼저 본다.
2. 바뀌었다면 `docs/status.md`를 우선 갱신한다.
3. 원칙이 바뀌었는지 보고, 바뀌었다면 `docs/개발방침.md`를 갱신한다.
4. 오래 유지할 핵심 결정이면 `docs/decisions.md`에 반영한다.
5. 최근 작업 흐름이 필요하면 `docs/logbook.md`에 짧게 남긴다.
6. 긴 상세 기록은 활성 문서에 쌓지 말고 `docs/archive/`로 보낸다.
