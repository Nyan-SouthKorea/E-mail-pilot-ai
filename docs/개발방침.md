# Email Agent - 개발 방침

> 이 문서는 오래 유지할 원칙만 적는다. 현재 상태와 세부 진행 상황은 `docs/status.md`를 기준으로 관리한다.

## 0. 작업 모드

### 0-1. 협업 모드

- 기본 모드는 `협업 모드`다.
- 사용자의 중간 질문, 선택, 피드백을 자주 반영하면서 진행한다.
- 이 모드에서도 모든 비사소한 작업 단위의 첫 게이트는 `docs/AGENT.md`다.
- 다음 큰 단계로 넘어갈 때는 사용자의 현재 의도를 더 자주 확인하는 편으로 운영한다.
- 이 모드에서 assistant는 사용자에게 보내는 답변 앞에 기본적으로 `협업)` prefix를 붙인다.

### 0-2. 에이전트 모드

- 사용자가 명시적으로 `에이전트 모드`를 선언하면, 그 시점부터는 문서에 기록된 목표와 규칙을 기준으로 스스로 이어서 진행한다.
- 이 모드에서는 사용자가 자리를 비웠을 가능성을 기본 전제로 둔다.
- 명시적인 중단 지시가 없는 한 이미 시작한 수집, 동기화, 백필, 분류, 보고, 배포 준비 작업을 끊지 않는다.
- 장시간 다단계 파이프라인은 가능하면 현재 대화 세션이나 단일 터미널에만 의존하지 않게 만든다.
- `10분 이상` 걸릴 가능성이 있거나 `여러 단계가 자동으로 이어지는` 작업은 watchdog 대상으로 본다.
- 이 모드에서 assistant는 사용자에게 보내는 답변 앞에 기본적으로 `AI)` prefix를 붙인다.

### 0-3. 안내 말투

- 사용자에게 설명하거나 진행 상황을 알릴 때는 친절하고 차분한 말투를 유지한다.
- 다만 친절함 때문에 판단이나 상태를 흐리게 쓰지 않는다.
- 즉 톤은 부드럽게 유지하되, 현재 상태, 위험, 다음 단계는 분명하게 전달한다.
- 계획, 구조, 다음 작업을 설명할 때는 먼저 쉬운 말로 큰 그림을 설명한다.
- 기술 용어가 필요하면 바로 쓰기보다, 먼저 쉬운 표현을 주고 그다음에 용어를 붙인다.
- 사용자가 이해하기 어렵다고 말하면 같은 내용을 더 쉬운 말로 다시 풀어 설명하는 것을 기본 행동으로 둔다.

### 0-4. 전문가적 방향 제시

- 사용자가 두서 없이 요청하거나 비효율적인 순서로 작업을 지시할 수 있다는 점을 기본 전제로 둔다.
- 이런 경우 요청을 그대로 세부 실행으로 옮기기보다, 무엇이 비효율적인지 짚고 더 적절한 큰 그림과 권장 순서를 먼저 제시한다.
- 즉 사용자의 목표는 존중하되, 실행 순서와 작업 단위는 더 좋은 방향으로 재구성해서 안내한다.
- 단, 사용자의 의도를 무시하거나 과도하게 우회하지는 않고, 왜 방향을 조정하는지 근거를 함께 설명한다.

## 1. 프로젝트 방향

- 목표는 이메일을 받아 구조화된 업무 결과로 바꾸는 재사용 가능한 자동화 스택을 만드는 것이다.
- 현재 구현 순서는 `mailbox -> analysis -> exports`다.
- 초반 기본 전달물은 `이메일 수신 -> 분류/추출 -> Excel 출력` 경로다.
- 개발 단계에서는 Python으로 직접 실행 가능해야 하고, 배포 단계에서는 Windows `exe`로 패키징 가능한 형태를 함께 지향한다.
- 초기 사용자 경험은 CLI보다 GUI 기반 개인용 데스크톱 앱을 우선한다.
- 메일 연동 설정은 사용자가 이메일 주소와 인증 정보만 입력하고, 나머지 서버 설정은 앱이 자동 탐지해 제안하는 흐름을 우선한다.
- 사용자 프로필 폴더 안의 예시 메일과 기대 산출물은 `참고자료` 아래 reference fixture로 취급하고, 실제 런타임 산출물은 같은 프로필의 `실행결과` 아래로 분리한다.
- 현재 보이는 샘플 메일 형식에 설계를 맞추지 않고, 실제 업무 메일의 비정형성과 멀티모달 입력을 기본 전제로 둔다.
- 답장 자동화와 외부 알림은 위 기본 경로가 안정화된 뒤 점진적으로 붙인다.

## 2. 핵심 개발 원칙

### 2-1. 구조화 출력 우선

- LLM 응답은 자유서술보다 schema 기반 구조화 출력을 우선한다.
- 분류, 추출, 요약, 답장 draft 모두 가능한 한 명시적 필드 계약을 먼저 정한다.

### 2-1-1. LLM 비용은 기록하되 설계 기준은 성능과 정확도로 둔다

- OpenAI usage와 예상 비용은 계속 로그로 남기고 추적한다.
- 하지만 비용 절약 자체를 이유로 LLM 사용을 과도하게 아끼지는 않는다.
- 해석, 요약, 비정형 문서 이해, 멀티모달 추출처럼 LLM이 더 잘하는 일은 적극 사용한다.
- 반대로 행 위치 계산, 수식 보존, 접속 성공 판정, idempotent key 계산처럼 결정적 코드가 더 잘하는 일은 rule/code 경로가 맡는다.
- 즉 판단 기준은 `싸게`가 아니라 `더 정확하고 더 잘 작동하는가`다.

### 2-2. idempotent 처리 우선

- 같은 이메일을 여러 번 읽어도 중복 row, 중복 답장, 중복 알림이 생기지 않는 구조를 우선한다.
- mailbox cursor, message id, thread id, export row key, action log를 일관된 기준으로 남긴다.

### 2-3. 사람 승인 경계는 명확하게 둔다

- 자동 reply나 외부 발송 기능은 기본적으로 `draft -> review -> send` 경계를 먼저 만든다.
- 초기 단계에서는 `무조건 자동 발송`보다 `안전한 draft 생성`을 우선한다.

### 2-4. 민감정보와 원문 보호

- 실제 이메일 원문, 첨부파일, API 키, 계정 정보는 공개 문서에 적지 않는다.
- 개발 중 실제 프로필 기반 기본값은 리포 바깥 `../secrets/사용자 설정/<이름>/참고자료`, `../secrets/사용자 설정/<이름>/실행결과` 계층을 사용해 reference와 runtime을 분리한다.
- `results/`는 프로필이 아직 정해지지 않은 일회성 실험이나 재생성 가능한 범용 결과를 둘 때만 보조적으로 사용한다.
- 단, 배포본이나 로컬 실행형 프로필 파일은 예외적으로 실행 경로 인접의 런타임 파일을 허용한다.

### 2-5. 단순한 모듈 인터페이스

- 각 요소기술은 독립 폴더와 공통 진입점으로 관리한다.
- provider나 backend가 여러 개여도 상위 사용법은 하나로 유지한다.
- 데모, 실험, 운영 코드는 가능한 한 같은 파일에 섞지 않는다.

### 2-6. 오버엔지니어링 금지

- 처음부터 과도한 추상화, 미사용 옵션, 추측성 fallback을 넣지 않는다.
- 먼저 주경로를 단순하게 완성하고, 실제 문제를 본 뒤 필요한 만큼만 확장한다.

### 2-7. 코드 스타일 단순화

- 좋은 기능을 오버엔지니어링 없이 재사용성과 유지보수성이 좋은 코드로 구현하는 것을 우선한다.
- 함수와 클래스 중 무엇을 쓸지는 읽기 편의만이 아니라 책임 분리, 재사용성, 변경 용이성을 기준으로 판단한다.
- 상태와 동작을 함께 관리하는 도메인 개념은 class와 객체지향 설계를 적극 사용할 수 있다.
- 함수 시그니처는 현재 단계에서 지나치게 복잡하게 만들지 않는다.
- 모든 공개 함수와 핵심 메서드는 한국어 docstring으로 `기능 / 입력 / 반환`을 기본적으로 적는다.
- 표준 라이브러리 수준의 객체지향 문법과 보조 문법은 재사용성과 보일러플레이트 감소에 도움이 되면 사용한다.
- 단, metaclass, descriptor, 과도한 magic method override, 과도한 typing 문법처럼 이해 비용이 높은 문법은 꼭 필요한 근거가 있을 때만 사용한다.

### 2-8. 계획 기반 실행

- 작업은 시작 전에 짧은 계획을 세우고 순서대로 진행한다.
- 중간에 계획이 바뀌면 이유를 남긴다.
- 장시간 실행, 다단계 자동화, 대량 백필처럼 나중에 컨텍스트가 많이 압축될 수 있는 작업은 실행 전에 계획 문서를 남긴다.
- 사용자의 요청이 현재 단계에서 비효율적이면, 그 요청을 기준으로 바로 계획을 세우지 않고 먼저 더 나은 작업 순서를 제안한 뒤 그 기준으로 계획을 세운다.

### 2-8-1. 권한이나 비밀번호가 필요한 더 좋은 경로가 있으면 먼저 사용자에게 묻는다

- 시스템 권한, sudo 비밀번호, 외부 서비스 자격 증명처럼 사용자가 제공해야만 하는 정보가 있으면 먼저 그 사실을 분명히 알린다.
- 우회 경로가 가능하더라도, 더 표준적이거나 안정적인 경로가 권한 제공에 달려 있다면 우선 사용자의 확인이나 비밀번호 제공 여부를 먼저 묻는다.
- 사용자가 권한 제공을 원하지 않거나 바로 제공할 수 없을 때만, 그다음 선택지로 안전한 우회 경로를 제안하고 실행한다.
- 즉, 무조건 멈추는 것도 아니고 무조건 우회하는 것도 아니라, 먼저 정석 경로를 설명하고 사용자 선택을 받은 뒤 진행한다.

### 2-9. 장시간 실행 로그

- 1분 이상 걸릴 수 있는 실행은 진행 상태를 주기적으로 출력한다.
- 재실행이 비싼 작업은 가능하면 부분 재실행과 중간 상태 확인을 지원한다.
- 긴 명령이 돌 때는 가능한 범위에서 핵심 터미널 출력도 함께 요약해 사용자에게 알린다.
- watchdog를 붙일 장시간 작업은 실전 실행 전에 sample 코드와 smoke 결과를 먼저 만든다.

### 2-10. 디렉토리 구조도 주기적으로 정리한다

- 코드, 문서, 템플릿, 예시, 생성 결과가 한 디렉토리에 뒤섞이지 않도록 주기적으로 구조를 점검한다.
- 새 파일을 만들 때도 처음부터 `런타임 코드`, `실험 코드`, `도구`, `문서`, `예시`, `생성 산출물`의 역할을 구분한다.
- 아직 구현을 시작하지 않은 미래 기능용 빈 디렉토리와 플레이스홀더 README는 미리 늘리지 않는다.
- 공용 모듈도 실제로 둘 이상의 활성 경로가 공유할 때만 추출한다.

### 2-11. 문서 refresh를 작업 시작 게이트로 둔다

- 비사소한 작업에서는 시작 게이트를 반드시 다시 통과한다.
- 특히 `에이전트 모드`에서는 새 단계 착수 전, 결과 정리 직전, 후속 실행 시작 직전에 다시 `docs/README.md`와 `docs/status.md`를 읽는다.

### 2-12. 진행 보고는 짧고 자주 한다

- 장시간 작업에서는 진행 보고 빈도를 낮추지 않는다.
- 보고 내용은 기본적으로 `무엇을 확인했는지 / 어디서 막혔는지 / 다음에 무엇을 할지`를 포함한다.
- 사용자는 터미널 원문을 보지 못하므로, 긴 실행에서는 핵심 출력과 의미를 짧게 번역해 같이 전달한다.
- 실제로 돌고 있는 작업을 말할 때는 반드시 `ps`, `tty`, 진행률 파일, 상태 파일 같은 근거를 먼저 확인한다.
- 이런 진행 보고도 친절한 안내 말투를 유지하되, 사실 관계와 판단 기준은 직접적으로 적는다.

### 2-13. 장시간 작업은 안정된 마일스톤마다 commit을 남긴다

- 장시간 작업은 끝날 때만 한 번에 정리하지 말고, 안정된 마일스톤마다 local commit을 기본적으로 남긴다.
- push는 자동 상시 동작으로 두지 않지만, 원격에 남길 가치가 있는 안정된 마일스톤이면 같은 흐름에서 함께 점검한다.
- commit 메시지는 기본적으로 한국어 중심으로 쓴다.

### 2-14. background job은 detached 기준으로 본다

- polling, backfill, bulk export, 큐 worker처럼 장시간 도는 작업은 단순한 포그라운드 셸 루프보다 `pid`, `status`, `events`를 남기는 detached supervisor를 기본값으로 본다.
- 장시간 작업의 최소 산출물은 `job spec`, `pid`, `status.local.md`, `events.log`, `stdout/stderr`, `종료 코드 또는 종료 원인`이다.

### 2-15. Python 실행과 Windows exe 패키징을 함께 고려한다

- 개발 중 기본 실행 형태는 Python이지만, 최종 사용자 배포는 Windows `exe`까지 포함해 설계한다.
- 따라서 주경로에는 Linux 전용 의존성, 셸 스크립트에 강하게 묶인 진입점, OS 고정 경로 가정을 가능하면 넣지 않는다.
- 실행 진입점, 설정 파일, 템플릿, 리소스 경로는 패키징 이후에도 유지될 수 있게 분리한다.

### 2-15-1. Python 의존성은 프로젝트 전용 venv에 설치한다

- Python 패키지는 시스템 Python에 직접 설치하지 않는다.
- 기본 개발 환경은 워크스페이스의 `envs/venv` 같은 프로젝트 전용 가상환경 하나를 기준으로 맞춘다.
- 새 의존성이 필요하면 먼저 해당 venv에 설치하고, 이후 코드와 문서도 그 기준에 맞춘다.
- 이렇게 해야 시스템 환경 오염을 줄이고, 이후 재현과 패키징 기준을 단순하게 유지할 수 있다.

### 2-16. GUI 기반 프로필 구성을 기본 사용 흐름으로 본다

- 초기 사용자 흐름은 `프로필 생성 -> 이메일 계정/설정 입력 -> 저장 -> 실행` 순서를 기본값으로 둔다.
- 프로필에는 최소한 이메일 ID, 비밀번호 또는 앱 비밀번호, provider 관련 설정, 실행 옵션을 담을 수 있게 설계한다.
- 프로필 데이터는 초기 버전에서 로컬 `json` 같은 단순 포맷으로 저장하는 것을 허용한다.
- 배포본에서는 개인용 PC 전제를 바탕으로 실행 파일 인접 디렉토리에 프로필 파일을 두는 휴대형 구조를 우선 고려한다.
- 다만 실제 프로필 파일은 코드나 문서와 같은 Git 추적 자산이 아니라 로컬 런타임 자산으로 취급한다.

### 2-17. 메일 설정 탐지는 rule/probe-first로 설계한다

- 프로필 생성 단계에서 사용자는 이메일 주소와 인증 정보만 입력하는 것을 기본으로 둔다.
- 앱은 도메인 규칙, provider 프리셋, 표준 autodiscover/autoconfig, common host/port/security probe, 실제 접속 테스트 순으로 IMAP / POP / SMTP 후보를 탐지한다.
- 성공했거나 신뢰도 높은 설정값은 GUI에 자동으로 채우고, 사용자가 확인하거나 수정할 수 있게 한다.
- LLM은 설명 생성, 예외 provider 힌트 보조 추론, fallback 정리에만 제한적으로 사용하고, 실제 접속 가능 여부 판정의 주수단으로 삼지 않는다.

### 2-18. reference fixture는 읽기 전용으로 취급한다

- `secrets/사용자 설정/<이름>/참고자료/...` 아래의 예시 이메일, 첨부파일, 기대 산출물은 개발 기준과 비교용 fixture다.
- assistant와 런타임 코드는 이 reference 자산을 덮어쓰거나 정리본으로 재저장하지 않는다.
- 실제 수신 메일과 생성 산출물은 `secrets/사용자 설정/<이름>/실행결과/...`에 저장하고, fixture는 회귀 테스트와 품질 비교 기준으로만 사용한다.

### 2-19. 입력 해석은 multimodal-first로 설계한다

- 실제 신청 메일은 본문 텍스트, HTML, inline 이미지, 스크린샷, 스캔 PDF, 첨부 이미지, ZIP 안 문서가 섞여 들어올 수 있다고 가정한다.
- 본문 텍스트가 충분하지 않더라도 이미지 안 텍스트와 표, 문서 첨부 안의 핵심 정보를 읽어야 한다.
- 따라서 입력 계층은 `plain text parser`만으로 닫지 않고, OCR/VLM/table extraction을 붙일 수 있는 구조로 둔다.
- 현재 fixture 2건은 smoke와 회귀 테스트에만 쓰고, 이 형식에 과적합되는 분기 로직을 만들지 않는다.

### 2-20. 수신 이메일은 raw 보존과 human preview를 분리한다

- 원본 보존 포맷은 MIME 구조를 유지하는 `.eml`을 기본 canonical 원본으로 둔다.
- 사용자가 편하게 열어보는 용도는 `.html` preview를 기본으로 둔다.
- inline 이미지와 첨부파일은 별도 assets/attachments 경로로 풀어, preview와 연결 가능하게 한다.
- PDF는 공유나 출력용으로는 유용할 수 있지만, 원본 보존과 재파싱 기준 포맷으로는 두지 않는다.
- 메일별 관리 단위는 최소 `raw.eml`, `preview.html`, `attachments/`, `normalized.json`, `summary.md`를 포함하는 번들로 본다.
- 실제 기본 저장 위치는 `secrets/사용자 설정/<이름>/실행결과/받은 메일/<bundle-id>/`로 둔다.
- 사용자가 직접 보는 상위 폴더는 한국어 이름을 쓰고, bundle id 같은 기계 관리용 하위 식별자는 ASCII 기반으로 만든다.

### 2-21. Excel 산출물은 human-first append와 스타일 보존을 따른다

- 산출물 Excel은 사람이 언제든 직접 수정할 수 있는 문서로 전제한다.
- AI는 기존 사람이 작성한 내용이나 위쪽 행을 임의 재정렬하거나 덮어쓰기보다, 마지막 사용 행 다음에 append하는 방식을 기본으로 둔다.
- 기존 폰트, 정렬, 줄바꿈, 셀 너비, 수식, 서식 규칙은 템플릿 행이나 인접 행을 기준으로 최대한 이어받는다.
- workbook 갱신은 기계가 새 문서를 만드는 느낌보다, 사람이 이어서 작성한 것처럼 자연스러운 결과를 목표로 한다.

### 2-22. 요약은 신청서 기반으로 하되 중복을 줄이고 가독성을 높인다

- 요약은 신청서와 이메일 본문, 필요한 경우 첨부 소개 자료를 참고해 생성한다.
- 단순 원문 복붙보다 중복 표현 제거, 구체성 유지, 짧은 문장화, 보고용 가독성 향상을 우선한다.
- `기업소개(한줄)`, `사업내용 요약`, `요청사항 요약`처럼 사람이 바로 훑을 수 있는 필드는 압축 품질을 중요하게 본다.

### 2-23. 입력과 출력 사이의 데이터 계약은 4층으로 유지한다

- 메일 수집 보관 단위는 `MailBundle`, 후속 해석 입력은 `NormalizedMessage`, 분석 산출물은 `ExtractedRecord`, Excel 직전 계약은 `ExportRow`로 둔다.
- 각 계층은 자기 바로 다음 계층에 필요한 정보만 책임지고, provider별 raw payload나 workbook 내부 구현 세부사항을 불필요하게 새지 않게 한다.
- `MailBundle`은 원본 보존과 artifact inventory를 책임지고, `NormalizedMessage`는 분석 공통 입력을 책임지며, `ExtractedRecord`는 값과 근거를 함께 들고 다닌다.
- 이렇게 중간 계약을 나누어야 fixture 기반 smoke, 실제 메일 연동, 나중의 GUI/배포 경로가 같은 데이터 기준을 재사용할 수 있다.

### 2-24. Excel 출력은 전역 고정 양식보다 프로필별 템플릿 해석을 우선한다

- 프로젝트 전체에서 하나의 고정 Excel 열 구조를 먼저 박아두지 않는다.
- 사용자 프로필 폴더의 레퍼런스 Excel 문서를 읽어, 시트/헤더/열 의미/스타일 규칙을 프로필별 템플릿으로 해석한다.
- `ExtractedRecord`는 프로필과 무관한 공통 의미 필드를 유지하고, `ExportRow`는 그 공통 필드를 프로필별 템플릿에 맞게 재배열한 결과로 본다.
- LLM은 템플릿 열 이름의 의미 해석, 공통 필드와의 매핑 보조, 긴 원문 요약/정리에 사용한다.
- 실제 셀 위치 계산, append 행 결정, 스타일 복사, 수식 보존, 파일 저장은 결정적 코드 경로가 책임진다.

## 3. 문서 운영 원칙

### 3-1. 문서 역할은 하나씩만 가진다

| 문서 | 역할 |
|---|---|
| `README.md` | 레포 소개와 진입점 |
| `docs/README.md` | 문서 허브 |
| `docs/status.md` | 현재 상태와 다음 작업의 단일 기준 |
| `docs/decisions.md` | 현재 유효한 핵심 결정 |
| `docs/logbook.md` | 최근 작업 로그 |
| `docs/archive/` | 오래된 상세 기록 |

- 같은 상태 설명을 여러 상위 요약 문서에 반복하지 않는다.
- 상태가 바뀌면 우선 `docs/status.md`를 갱신한다.
- 원칙이 바뀌었을 때만 `docs/개발방침.md`를 갱신한다.
- 새 에이전트나 새 세션의 시작 절차는 `docs/README.md`를 기준으로 통일한다.

### 3-2. 상위 문서는 얇게 유지한다

- 루트 `README.md`에는 프로젝트 소개, 모듈 링크, 문서 진입점만 둔다.
- `docs/README.md`는 색인과 역할 설명만 둔다.
- 현재 단계에서 보조 개요 문서는 꼭 필요할 때만 추가한다.

### 3-3. 결정과 로그는 분리한다

- `docs/decisions.md`에는 현재도 유효한 프로젝트 결정만 남긴다.
- 세부 과정, 시행착오, 실행 기록은 `docs/logbook.md` 또는 `docs/archive/`로 보낸다.

### 3-4. 최근 로그만 활성 문서에 둔다

- `docs/logbook.md`는 최근 세션만 유지한다.
- 문서가 길어지거나 현재 단계와 직접 관련 없는 항목이 많아지면 `docs/archive/`로 옮긴다.

### 3-5. 생성 산출물과 사람이 유지하는 문서를 분리한다

- code-generated CSV, JSON, XLSX, HTML, 로그 파일은 공식 문서가 아니다.
- 자동 생성 결과물은 기본적으로 `secrets/사용자 설정/<이름>/실행결과/` 아래에 두고, 사람이 계속 읽을 필요가 있는 경우에만 요약을 문서로 승격한다.

### 3-6. 환경 문서는 별도 유지한다

- 설치 절차, venv 경로, 검증 명령은 관련 모듈 문서에서 관리한다.
- 상위 문서에는 환경 문서 링크만 두고 설치 명령을 반복하지 않는다.

### 3-7. 민감 정보는 공개 문서에 쓰지 않는다

- 실제 계정, 키, 메일 원문, 첨부파일, 내부 주소, 사용량 로그 같은 운영 정보는 리포 바깥 로컬 문서에서만 관리한다.
- 단, 배포본 실행을 위한 로컬 프로필 파일은 예외적으로 사용자 PC의 실행 파일 인접 경로에 둘 수 있지만, 이 역시 Git에 포함하지 않는다.
- reference fixture로 둔 예시 메일과 기대 산출물도 원본 훼손 없이 유지한다.

### 3-8. 누적 보고서 문서는 날짜 prefix로 정렬 가능하게 만든다

- `README`, `status`, `개발방침`, `decisions`, `logbook`처럼 항상 최신 기준을 가리켜야 하는 문서는 고정 파일명을 유지한다.
- 누적되는 조사/환경/보고서는 `YYMMDD_HHMM_한국어제목.md` 형식을 기본으로 쓴다.

## 4. 자산과 데이터 운영 원칙

- 실제 고객 메일, 대용량 첨부파일, 임시 export, 로컬 DB는 기본적으로 리포에 포함하지 않는다.
- 다만 재현에 꼭 필요한 소형 예제나 redacted sample은 예외적으로 포함할 수 있다.
- 문서용 스크린샷, 다이어그램, 샘플 이미지는 `docs/assets/` 아래에서 정리한다.

### 4-1. 로컬 워크스페이스 구조

- 로컬 워크스페이스는 한 루트 아래 `repo/`, `envs/`, `results/`, `secrets/`를 sibling으로 둔다.
- `repo/`는 Git 추적 대상 코드와 문서만 둔다.
- `envs/`는 로컬 가상환경만 둔다.
- `results/`는 프로필 미지정 실험 결과나 범용 임시 산출물을 둔다.
- `secrets/`는 Git에 넣을 수 없는 로컬 전용 운영 정보만 둔다.
- 실제 사용자 프로필 기준 reference/runtime은 `secrets/사용자 설정/<이름>/참고자료/`, `secrets/사용자 설정/<이름>/실행결과/`를 표준 구조로 둔다.

### 4-2. 추적 자산과 로컬 자산을 구분한다

- 작은 템플릿이나 redacted sample은 리포에 포함할 수 있다.
- 실제 메일 데이터, 첨부파일 원본, 대량 export 결과, 인증 캐시는 로컬 경로 또는 gitignore 경로에 둔다.

## 5. 문서를 갱신할 때의 체크리스트

1. 이번 변경의 현재 상태 기준이 바뀌었는지 먼저 본다.
2. 바뀌었다면 `docs/status.md`를 우선 갱신한다.
3. 원칙이 바뀌었는지 보고, 바뀌었다면 `docs/개발방침.md`를 갱신한다.
4. 오래 유지할 핵심 결정이면 `docs/decisions.md`에 반영한다.
5. 최근 작업 흐름이 필요하면 `docs/logbook.md`에 짧게 남긴다.
6. 긴 상세 기록은 활성 문서에 쌓지 말고 `docs/archive/`로 보낸다.
