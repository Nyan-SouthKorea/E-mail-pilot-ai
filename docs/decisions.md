# Active Decisions

> 현재도 유효한 핵심 결정만 유지한다.

## 2026-03-21 | 문서 운영은 단일 역할 기준으로 유지한다

- 결정: 현재 상태는 `status`, 안정된 원칙은 `개발방침`, 최근 작업은 `logbook`, 오래된 상세 이력은 `archive`로 분리한다.
- 이유: 상위 요약 문서가 겹치면 쓰는 비용이 커지고 최신 기준이 흐려진다.

## 2026-03-21 | 모든 비사소한 작업은 `docs/AGENT.md`에서 다시 시작한다

- 결정: 코드 수정, 문서 수정, 커밋, 푸시, 아키텍처 판단 같은 비사소한 작업은 항상 `docs/AGENT.md`를 먼저 읽고, 그 문서가 지시하는 `docs/README.md -> docs/status.md` 순서를 거친 뒤 진행한다.
- 이유: 세션이 길어지거나 컨텍스트가 압축돼도 방향성을 다시 맞출 수 있고, 불필요한 작업을 줄일 수 있다.

## 2026-03-21 | 기본 작업 모드는 `협업 모드`다

- 결정: 기본값은 `협업 모드`로 두고, 사용자가 명시적으로 선언하면 `에이전트 모드`로 전환한다.
- 이유: 이 프로젝트는 아직 요구사항을 함께 좁혀야 하는 초기 단계이기 때문이다.

## 2026-03-21 | v1 기본 전달 경로는 `이메일 수신 -> 분석 -> Excel 출력`이다

- 결정: 자동 답장과 외부 알림은 중요한 확장 범위지만, v1의 우선 구현 경로는 구조화 분류와 Excel 출력으로 둔다.
- 이유: 가장 빠르게 업무 가치가 보이고, 데이터 계약과 action policy를 안전하게 고정하기 쉽다.

## 2026-03-21 | 답장 자동화는 기본적으로 `draft 우선` 정책으로 시작한다

- 결정: reply 기능은 초기부터 자동 발송을 기본값으로 두지 않고, 먼저 draft 생성과 승인 경계를 설계한다.
- 이유: 오분류나 잘못된 추출이 외부 커뮤니케이션으로 바로 이어지는 리스크를 줄여야 한다.

## 2026-03-21 | 로컬 워크스페이스는 `repo / envs / results / secrets` sibling 구조를 지향한다

- 결정: Git 추적 대상은 `repo/`, 로컬 가상환경은 `envs/`, 재생성 가능한 실행 결과는 `results/`, 민감 정보는 `secrets/`에서 관리한다.
- 이유: 코드와 문서, 로컬 실행 상태, 민감 데이터를 분리해야 운영이 단순해진다.

## 2026-03-21 | 장시간 pipeline은 detached supervisor와 smoke-first 기준을 따른다

- 결정: polling daemon, backfill, bulk export, queue worker 같은 장시간 작업은 가능하면 detached supervisor로 분리하고, 실전 실행 전에 정상 종료 / 실패 종료 / stale 상태를 검증하는 smoke를 먼저 통과시킨다.
- 이유: 세션 종료나 중간 대화와 무관하게 작업 연속성을 유지하려면, 실제 프로세스와 상태 파일 기준으로 복구 가능한 구조가 필요하다.

## 2026-03-21 | 초기 루트 구조는 현재 활성 범위만 남긴다

- 결정: 현재 루트에는 `mailbox`, `analysis`, `exports`, `llm`만 active 디렉토리로 두고, workflow/reply/notification/shared/tests/tools/examples 같은 미래 확장용 구조는 실제 구현이 시작될 때 추가한다.
- 이유: 아직 쓰지 않는 빈 디렉토리와 플레이스홀더 문서를 미리 늘리면 현재 범위가 흐려지고, 중복 기록과 오버엔지니어링이 빠르게 커지기 때문이다.

## 2026-03-21 | 사용자 안내 말투는 친절하고 명확하게 유지한다

- 결정: 사용자에게 설명, 진행 보고, 상태 안내를 할 때는 친절하고 차분한 말투를 유지하되, 사실 관계와 판단은 모호하지 않게 명확히 전달한다.
- 이유: 사용감은 부드럽게 유지하면서도, 장시간 작업과 기술 판단에서 필요한 정보 밀도와 정확성을 잃지 않기 위해서다.

## 2026-03-21 | 사용자의 요청이 비효율적이면 더 나은 큰 그림을 먼저 제시한다

- 결정: 사용자가 비효율적인 순서나 과도하게 세부적인 방향으로 요청하더라도, 그 요청을 그대로 따라가기보다 무엇이 비효율적인지 설명하고 더 적절한 큰 그림과 권장 순서를 먼저 제시한다.
- 이유: 이 프로젝트는 사용자의 목표는 분명하지만 구현 순서와 시스템 설계는 계속 바뀔 수 있으므로, 전문가 관점의 방향 정리가 없으면 재작업과 오버엔지니어링이 커지기 쉽다.

## 2026-03-21 | 실행 형태는 Python 직접 실행과 Windows exe 패키징을 함께 목표로 한다

- 결정: 개발 단계에서는 Python으로 직접 실행 가능한 구조를 유지하고, 제품 배포 목표에는 Windows 환경에서 실행 가능한 `exe` 패키징을 포함한다.
- 이유: 사용자는 최종적으로 Python 개발 환경이 없는 Windows 사용자도 실행할 수 있는 형태를 원하며, 이 요구는 진입점 구조, 의존성 선택, 파일 경로 처리 방식에 직접 영향을 준다.

## 2026-03-21 | 기본 사용 방식은 GUI 프로필 기반 개인용 데스크톱 앱으로 둔다

- 결정: 초기 제품 사용 흐름은 CLI 직접 실행보다 GUI에서 프로필을 만들고, 저장된 프로필 설정으로 자동화를 실행하는 데스크톱 앱 형태를 우선한다.
- 이유: 사용자는 개인용 PC에서 쉽게 설정하고 반복 실행할 수 있는 형태를 원하며, 이 요구는 진입점 설계, 설정 저장 방식, 배포 구조에 직접 영향을 준다.

## 2026-03-21 | 초기 프로필 설정은 실행 파일 인접 로컬 JSON 저장을 허용한다

- 결정: 초기 버전에서는 이메일 ID, 비밀번호 또는 앱 비밀번호 등 프로필 설정을 실행 파일 인접 디렉토리의 로컬 `json` 파일로 저장하는 방식을 허용한다.
- 이유: 현재 우선순위는 보안 고도화보다 개인용 PC에서의 단순한 설정과 실행 경험이며, 사용자는 이 프로젝트를 우선 개인용 환경에서 운용할 계획이기 때문이다.

## 2026-03-21 | 프로필 입력은 최소 정보만 받고 메일 설정은 자동 보정한다

- 결정: 초기 GUI 프로필 생성에서는 사용자가 이메일 주소와 인증 정보만 입력하도록 하고, 앱이 protocol / host / port / security 같은 메일 설정 후보를 자동 탐지해 GUI와 프로필 설정값에 함께 반영한다.
- 이유: 사용자가 IMAP, POP, SMTP와 provider별 서버 설정을 직접 아는 것을 전제로 하면 설정 장벽이 커지고, 개인용 데스크톱 앱 방향과도 맞지 않기 때문이다.

## 2026-03-21 | 메일 설정 탐지는 rule/probe-first, LLM fallback으로 둔다

- 결정: 메일 설정 탐지는 도메인 규칙, provider 프리셋, autodiscover/autoconfig, 접속 테스트를 먼저 사용하고, LLM은 예외 케이스 보조 추론과 설명 생성에만 제한적으로 사용한다.
- 이유: 실제 연결 가능 여부는 결정적이고 테스트 가능한 방식으로 판정해야 하며, 설정 탐지를 LLM 중심으로 두면 비용과 불확실성이 함께 커지기 때문이다.

## 2026-03-21 | 사용자 설정 폴더의 예시 입력과 기대 산출물은 read-only reference로 둔다

- 결정: `secrets/사용자 설정/<이름>/...` 아래의 예시 메일, 첨부파일, 기대 산출물은 런타임 수정 대상이 아니라 reference fixture로만 사용한다.
- 이유: 사용자는 이 폴더를 프로필별 보안 정보와 레퍼런스 보관용으로 쓰려 하며, 실제 생성 산출물이 여기를 덮어쓰면 비교 기준과 운영 기준이 동시에 흐려진다.

## 2026-03-21 | 수신 이메일의 canonical 보관 포맷은 EML, 열람 포맷은 HTML preview로 둔다

- 결정: 실제 수신 메일은 `.eml`을 canonical 원본으로 저장하고, 사용자가 편하게 열어보는 용도는 `preview.html`을 기본 파생본으로 생성한다.
- 이유: `.eml`은 헤더와 MIME 구조, 첨부 관계를 가장 충실하게 보존하고, `.html`은 사람이 내용을 빠르게 열람하기에 가장 편하기 때문이다. PDF는 선택적 파생본이면 충분하다.

## 2026-03-21 | 메일별 산출물은 하나의 관리 번들로 남긴다

- 결정: 실제 수신 메일마다 최소 `raw.eml`, `preview.html`, `attachments/`, `normalized.json`, `summary.md`를 같은 관리 단위로 남긴다.
- 이유: 원본 보존, 사람 열람, 기계 처리 결과, 요약 산출물을 한 묶음으로 남겨야 추적과 재처리가 쉬워진다.

## 2026-03-21 | Excel 산출물은 human-first append와 스타일 보존을 따른다

- 결정: AI는 사용자가 수정 가능한 Excel 문서를 전제로, 마지막 사용 행 다음에 append하는 것을 기본으로 하고 기존 폰트, 정렬, 줄바꿈, 수식, 서식을 최대한 이어받는다.
- 이유: 실제 운영 문서는 사람이 중간 수정과 보완을 계속하기 때문에, 기계가 문서의 톤과 구조를 깨면 실무 사용성이 크게 떨어진다.

## 2026-03-21 | 요약은 신청서 기반의 readable compression으로 만든다

- 결정: 엑셀 정리본의 요약 필드는 신청서 내용을 기초로 하되, 중복 표현을 줄이고 사람이 빠르게 읽을 수 있는 형태로 압축한다.
- 이유: 기대 산출물은 단순 추출이 아니라 보고와 검토에 적합한 가독성 높은 정리본이기 때문이다.

## 2026-03-21 | 입력 처리 설계는 현재 샘플 메일 2건에 과적합하지 않는다

- 결정: 현재 reference fixture 2건은 smoke와 회귀 비교용으로만 쓰고, 실제 설계 기준은 더 넓은 이메일 변형을 포괄하는 방향으로 둔다.
- 이유: 실제 신청 메일은 사용자가 AI 처리를 상정하지 않고 작성하기 때문에, 현재 샘플만 기준으로 잡으면 텍스트 없는 이미지 캡처, 이미지 속 표, 스캔 PDF 같은 입력에서 바로 깨질 가능성이 높다.

## 2026-03-21 | 이메일 입력은 multimodal extraction을 기본 전제로 둔다

- 결정: 이메일 해석은 plain text와 HTML만이 아니라 inline 이미지, 첨부 이미지, 스캔 PDF, ZIP 안 문서, 이미지 속 표까지 처리 가능한 multimodal 구조를 기본 전제로 둔다.
- 이유: 실제 업무 메일은 비정형 첨부와 시각 자료가 많고, 핵심 정보가 본문이 아니라 이미지나 문서 내부에 있는 경우가 흔하기 때문이다.

## 2026-03-21 | 안정된 마일스톤마다 local commit을 기본으로 남긴다

- 결정: 작업 결과가 안정된 마일스톤에 도달하면 local commit을 기본적으로 남기고, 원격에 남길 가치가 있는 마일스톤이면 push까지 같은 흐름에서 점검한다.
- 이유: 긴 세션에서 변경을 오래 작업 트리에만 쌓아두면 회귀 지점과 설명 책임이 흐려지므로, 마일스톤 단위 기록을 더 적극적으로 남기는 편이 협업에 유리하다.

## 2026-03-21 | 기본 데이터 흐름은 `MailBundle -> NormalizedMessage -> ExtractedRecord -> ExportRow`로 둔다

- 결정: 메일 원본 보관 단위는 `MailBundle`, 분석 공통 입력은 `NormalizedMessage`, 분석 산출물은 `ExtractedRecord`, Excel 직전 계약은 `ExportRow`로 나눈다.
- 이유: 메일 provider 구현, 멀티모달 입력 해석, LLM 추출, Excel 출력이 서로 다른 책임을 갖고 움직이기 때문에, 중간 계약이 없으면 구현이 빠르게 서로 얽히고 fixture 기반 검증도 어려워진다.

## 2026-03-21 | 주경로 코드는 객체지향과 재사용성을 허용하되 과한 추상화는 피한다

- 결정: 주경로 구현에서 class와 객체지향 설계를 배제하지 않고, 상태와 동작을 함께 관리해야 하는 도메인에는 적극 사용한다. 단, 오버엔지니어링이 되지 않도록 추상화 깊이와 미사용 유연성은 제한한다.
- 이유: 이 프로젝트는 기능과 방향이 계속 바뀔 가능성이 높아서, 단순한 가독성만보다 재사용성과 유지보수성이 더 중요하며, 그 목적에는 적절한 객체지향 설계가 유리하기 때문이다.

## 2026-03-21 | Excel 출력은 프로필별 레퍼런스 템플릿 기준으로 해석한다

- 결정: Excel 출력의 실제 열 구조와 시트 구성을 프로젝트 전체에서 하나로 고정하지 않고, 사용자 프로필 폴더의 레퍼런스 Excel 문서를 읽어 프로필별 템플릿 규칙으로 해석한다.
- 이유: 사용자마다 원하는 양식이 다를 수 있어서 열 이름, 순서, 시트 구조를 전역 고정하면 곧바로 재작업이 커지며, 더 안정적인 방법은 공통 의미 필드와 프로필별 템플릿을 분리하는 것이다.

## 2026-03-21 | LLM은 템플릿 의미 해석과 요약을 맡고 workbook 쓰기는 코드가 맡는다

- 결정: LLM은 레퍼런스 Excel 열의 의미를 해석하고 공통 필드와의 매핑을 보조하며, 긴 신청 내용을 양식에 맞게 요약하는 데 사용한다. 반면 실제 workbook의 행 위치 계산, 셀 쓰기, 스타일 복사, 수식 보존은 코드가 담당한다.
- 이유: 템플릿의 의미 해석은 비정형 이름과 사람 중심 양식을 다뤄야 해서 LLM이 유리하지만, 실제 파일 갱신은 재현성과 안정성이 중요해 코드가 맡는 편이 맞기 때문이다.
