# Active Decisions

> 현재도 유효한 핵심 결정만 유지한다.

## 2026-03-21 | 문서 운영은 단일 역할 기준으로 유지한다

- 결정: 현재 상태는 `status`, 안정된 원칙은 `개발방침`, 최근 작업은 `logbook`, 오래된 상세 이력은 `archive`로 분리한다.
- 이유: 상위 요약 문서가 겹치면 쓰는 비용이 커지고 최신 기준이 흐려진다.

## 2026-03-21 | 모든 비사소한 작업은 `docs/AGENT.md`에서 다시 시작한다

- 결정: 코드 수정, 문서 수정, 커밋, 푸시, 아키텍처 판단 같은 비사소한 작업은 항상 `docs/AGENT.md`를 먼저 읽고, 그 문서가 지시하는 `docs/README.md -> docs/status.md` 순서를 거친 뒤 진행한다.
- 이유: 세션이 길어지거나 컨텍스트가 압축돼도 방향성을 다시 맞출 수 있고, 불필요한 작업을 줄일 수 있다.

## 2026-03-21 | 기본 작업 모드는 `협업 모드`다

- 결정: 기본값은 `협업 모드`로 두고, 사용자가 명시적으로 선언하면 `에이전트 모드`로 전환한다.
- 이유: 이 프로젝트는 아직 요구사항을 함께 좁혀야 하는 초기 단계이기 때문이다.

## 2026-03-21 | v1 기본 전달 경로는 `이메일 수신 -> 분석 -> Excel 출력`이다

- 결정: 자동 답장과 외부 알림은 중요한 확장 범위지만, v1의 우선 구현 경로는 구조화 분류와 Excel 출력으로 둔다.
- 이유: 가장 빠르게 업무 가치가 보이고, 데이터 계약과 action policy를 안전하게 고정하기 쉽다.

## 2026-03-21 | 답장 자동화는 기본적으로 `draft 우선` 정책으로 시작한다

- 결정: reply 기능은 초기부터 자동 발송을 기본값으로 두지 않고, 먼저 draft 생성과 승인 경계를 설계한다.
- 이유: 오분류나 잘못된 추출이 외부 커뮤니케이션으로 바로 이어지는 리스크를 줄여야 한다.

## 2026-03-21 | 로컬 워크스페이스는 `repo / envs / results / secrets` sibling 구조를 지향한다

- 결정: Git 추적 대상은 `repo/`, 로컬 가상환경은 `envs/`, 재생성 가능한 실행 결과는 `results/`, 민감 정보는 `secrets/`에서 관리한다.
- 이유: 코드와 문서, 로컬 실행 상태, 민감 데이터를 분리해야 운영이 단순해진다.

## 2026-03-21 | 장시간 pipeline은 detached supervisor와 smoke-first 기준을 따른다

- 결정: polling daemon, backfill, bulk export, queue worker 같은 장시간 작업은 가능하면 detached supervisor로 분리하고, 실전 실행 전에 정상 종료 / 실패 종료 / stale 상태를 검증하는 smoke를 먼저 통과시킨다.
- 이유: 세션 종료나 중간 대화와 무관하게 작업 연속성을 유지하려면, 실제 프로세스와 상태 파일 기준으로 복구 가능한 구조가 필요하다.

## 2026-03-21 | 초기 루트 구조는 현재 활성 범위만 남긴다

- 결정: 현재 루트에는 `mailbox`, `analysis`, `exports`, `llm`만 active 디렉토리로 두고, workflow/reply/notification/shared/tests/tools/examples 같은 미래 확장용 구조는 실제 구현이 시작될 때 추가한다.
- 이유: 아직 쓰지 않는 빈 디렉토리와 플레이스홀더 문서를 미리 늘리면 현재 범위가 흐려지고, 중복 기록과 오버엔지니어링이 빠르게 커지기 때문이다.

## 2026-03-21 | 사용자 안내 말투는 친절하고 명확하게 유지한다

- 결정: 사용자에게 설명, 진행 보고, 상태 안내를 할 때는 친절하고 차분한 말투를 유지하되, 사실 관계와 판단은 모호하지 않게 명확히 전달한다.
- 이유: 사용감은 부드럽게 유지하면서도, 장시간 작업과 기술 판단에서 필요한 정보 밀도와 정확성을 잃지 않기 위해서다.
