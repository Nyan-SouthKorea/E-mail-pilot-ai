# 문서 허브

이 디렉토리는 역할이 겹치지 않는 최소 문서 구조를 유지하기 위한 허브다. 현재 상태는 `status`, 안정된 원칙은 `개발방침`, 최근 기록은 `logbook`, 핵심 결정은 `decisions`로 분리한다.

## 가장 중요한 규칙

- 비사소한 작업에서는 실행 시작 전, 코드 수정 전, 문서 수정 전, 결과 정리 전, 커밋 전, 푸시 전, 아키텍처 판단 전에 항상 시작 게이트를 다시 통과한다.
  - `협업 모드`, `에이전트 모드` 모두 항상 먼저 `docs/AGENT.md`를 읽고, 그 문서가 지시하는 순서대로 `docs/README.md -> docs/status.md`로 진입한다.
- `docs/README.md`를 읽은 뒤에는 예외 없이 바로 `docs/status.md`를 읽고 현재 목표, 현재 고정 기준, 현재 활성 모드를 먼저 확인한다.
- 그 다음에는 현재 작업과 직접 관련된 최소 범위의 모듈 문서만 읽고 작업한다.
- 현재 단계에 꼭 필요한 일만 수행하고, 과도한 추상화, 불필요한 자동화, 과도한 문서 생성을 하지 않는다.

## 새 에이전트 시작 절차

새 세션에서 에이전트에게는 기본적으로 아래 한 줄만 지시하면 된다.

`먼저 docs/AGENT.md를 읽고, 그 문서가 지시하는 순서대로 진행해.`

그 다음 절차는 아래로 고정한다.

1. `docs/status.md`를 먼저 읽고 현재 상태를 파악한다.
2. 이번 작업과 직접 관련된 모듈 `README.md`만 읽는다.
3. `docs/개발방침.md`를 읽고 그 규칙에 맞게 행동한다.
4. 필요할 때만 `docs/decisions.md`를 추가로 읽는다.
5. 과거 맥락이 꼭 필요할 때만 `docs/logbook.md`와 `docs/archive/`를 읽는다.

## Single Source of Truth

| 문서 | 역할 | 여기에 써야 하는 것 | 여기서 빼야 하는 것 |
|---|---|---|---|
| `README.md` | 레포 진입점 | 프로젝트 소개, 모듈 링크, 문서 진입점 | 상세 상태, 긴 이력 |
| `docs/status.md` | 현재 상태 기준 | 지금 완료된 것, 고정 기준, 다음 작업 | 오래된 실험 로그 |
| `docs/개발방침.md` | 안정된 원칙 | 개발/문서 운영 규칙 | 일시적인 진행 상황 |
| `docs/decisions.md` | 핵심 결정 | 현재도 유효한 의사결정 | 세세한 작업 메모 |
| `docs/logbook.md` | 최근 로그 | 최근 세션의 작업 흐름 | 오래된 전체 히스토리 |
| `<module>/README.md` | 모듈 진입점 | 역할, 상태, 구조 | 프로젝트 전체 상태 요약 |

## 추천 읽는 순서

1. `docs/status.md`
2. 관련 모듈 `README.md`
3. `docs/개발방침.md`
4. `docs/decisions.md`
5. 과거 맥락이 더 필요할 때만 `docs/logbook.md`와 `docs/archive/`

## 현재 문서 구조

```text
docs/
├── AGENT.md
├── README.md
├── status.md
├── 개발방침.md
├── decisions.md
└── logbook.md
```

- `docs/`에는 현재 활성 기준 문서만 남긴다.
- `archive/`, `assets/` 같은 보조 디렉토리는 실제 필요가 생길 때 만든다.

## 운영 규칙

- 같은 상태 설명을 여러 상위 요약 문서에 반복하지 않는다.
- 현재 상태를 바꿨다면 우선 `docs/status.md`를 갱신한다.
- 원칙이 바뀌었을 때만 `docs/개발방침.md`를 갱신한다.
- 중요 변경 뒤에는 필요한 문서만 최소 범위로 갱신한다. 습관적으로 모든 문서를 건드리지 않는다.
- 아직 쓰지 않는 미래 기능용 문서와 빈 디렉토리를 미리 만들지 않는다.
- 오래된 상세 기록은 필요할 때 `docs/archive/`로 내린다.
- 민감 정보와 실제 메일 원문은 공개 문서에 적지 않는다.
- 실데이터나 계정 정보가 필요한 실행 결과는 기본적으로 리포 바깥 로컬 경로에서 관리한다.
- 프로필 기반 reference와 runtime은 `secrets/사용자 설정/<이름>/참고자료/`, `secrets/사용자 설정/<이름>/실행결과/`를 기본 구조로 본다.
- 단, 배포본 또는 로컬 실행형 프로필 파일은 예외적으로 실행 경로 인접의 gitignored 런타임 위치를 허용한다.

## 문서 수정 체크리스트

1. 이번 변경의 최종 기준 문서가 `status`, `개발방침`, `README` 중 어디인지 먼저 정한다.
2. 같은 내용을 두 곳 이상에 쓰지 않고, 다른 문서에는 링크만 남긴다.
3. 실행 결과를 숫자로 기록할 때는 code-generated 산출물이나 계산식 기반 값만 사용한다.
4. 민감 데이터나 실제 고객 메일 내용이 섞였다면 리포 안 문서 대신 로컬 비공개 문서로 돌린다.
